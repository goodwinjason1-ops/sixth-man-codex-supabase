# QA Data Management

The app now has a resettable Lakers-style QA dataset for full end-user testing. It is not hardcoded into the user screens. It is generated by `scripts/manage-lakers-qa-data.mjs` and stored in Supabase through the Firestore-compatible `public.documents` table.

Every new QA row is tagged:

```json
{
  "qa": {
    "datasetId": "lakers-end-user-qa",
    "isTestData": true
  }
}
```

That tag is what makes cleanup safe.

## What The Dataset Covers

- All QA user roles: admin, president, vice president, coordinators, coaches, youth coach, player, parent, team manager, tryout assessor, pending user.
- Lakers teams, players, games, schedule, attendance, training plans, training records and match assessments.
- Coach voice note transcripts inside training and match assessment records.
- Individual development plans with parent visibility examples.
- Tryout sessions, tryout evaluations, scouting evaluations and selection decisions.
- Fair-play rotation data, shot events and advanced analytics source rows.
- Youth program records and operational rows such as invitations, scoring assignments and notifications.

## What Cleanup Touches

By default, cleanup removes only rows where:

```text
data.qa.datasetId == lakers-end-user-qa
```

It does not delete real club records, untagged rows, or QA auth users.

There are three optional cleanup flags:

- `--include-legacy-e2e`: removes the old untagged rows from `tests/e2eFixtures.js` by exact collection/id. Use a dry run first because old rows did not have a QA tag.
- `--include-sample-flags`: removes rows marked `isSampleData`, `isTestUser`, or `isTestData`.
- `--include-live-video-qa`: removes live video smoke-test uploads titled `Live QA Game Video...`, including native video session rows, recording rows, queued jobs and storage objects.

## Set Environment Variables

PowerShell:

```powershell
$env:SUPABASE_URL = "https://lipjgbcgwokhucbxinmn.supabase.co"
$env:SUPABASE_SERVICE_ROLE_KEY = "<service-role-key>"
$env:QA_PASSWORD = "<shared-qa-password>"
```

If you are logged in with the Supabase CLI, you can retrieve the service role key without printing it into chat:

```powershell
$json = npx supabase projects api-keys --project-ref lipjgbcgwokhucbxinmn -o json
$keys = $json | ConvertFrom-Json
$env:SUPABASE_SERVICE_ROLE_KEY = ($keys | Where-Object { $_.name -eq "service_role" } | Select-Object -First 1).api_key
```

Do not put `SUPABASE_SERVICE_ROLE_KEY` into frontend `.env` files, GitHub Pages variables, browser code or documentation.

## Safe Dry Run

Local dry run without credentials:

```powershell
npm run qa:data:dry-run
```

Remote dry run against Supabase:

```powershell
npm run qa:data:dry-run -- --include-legacy-e2e --include-sample-flags --include-live-video-qa
```

This prints how many rows would be seeded or removed. It does not change data.

## Reset To Fresh Lakers QA Data

Tagged QA data only:

```powershell
npm run qa:data:reset
```

Full reset after earlier untagged testing:

```powershell
npm run qa:data:reset -- --include-legacy-e2e --include-sample-flags --include-live-video-qa
```

This keeps the QA auth accounts available and reseeds fresh Lakers-style test data.

## Cleanup Without Reseeding

```powershell
npm run qa:data:cleanup
```

With legacy/sample/video cleanup:

```powershell
npm run qa:data:cleanup -- --include-legacy-e2e --include-sample-flags --include-live-video-qa
```

## Verify And Run Full End-User QA

Verify the expected tagged rows exist:

```powershell
npm run qa:data:smoke
```

Then run deployed live QA:

```powershell
$env:LIVE_QA_PASSWORD = "<shared-qa-password>"
npm run qa:live:deployed
```

The recommended sequence before a full end-user test is:

```powershell
npm run qa:data:dry-run -- --include-legacy-e2e --include-sample-flags --include-live-video-qa
npm run qa:data:reset -- --include-legacy-e2e --include-sample-flags --include-live-video-qa
npm run qa:live:deployed
```
